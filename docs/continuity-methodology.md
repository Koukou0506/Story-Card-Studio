# Phase C1 长篇连续性方法

本文定义 Story Card Studio 在长篇项目中的 Canon、状态、知情、剧情线、伏笔、时间线、偏差和项目健康规则。C1 只提供可追溯的候选、检查和人工处理流程，不自动修改角色卡、世界书、规划或采用正文。

## 1. Canon 权威层级

数值越小权威越高；冲突检查不得用低权威资料静默覆盖高权威资料。

| 等级 | 来源 | 默认 Canon 状态 |
|---:|---|---|
| 1 | 用户锁定 Canon | `locked` |
| 2 | 用户确认的正文事实 | `confirmed` |
| 3 | 当前采用正文中的明确事实 | `candidate`，确认后可升为 `confirmed` |
| 4 | 用户确认的角色卡或世界书 | `candidate` 或 `derived` |
| 5 | B1 当前采用规划 | `derived` |
| 6 | 用户确认的状态变化 / B2 入口出口 | `derived` |
| 7 | 未确认正文 Candidate Fact | `candidate` |
| 8 | 模型推断 | `candidate` |
| 9 | 模型建议或用户未确认假设 | `candidate` |

Canon 状态为 `locked / confirmed / derived / candidate / disputed / retconned / deprecated / unknown`。来源引用必须包含类型、ID、名称、字段、摘录、版本、权威、事实分类、锁定与有效性。没有真实来源的模型输出不能充当有效依据。

## 2. 候选事实确认流程

1. 从角色卡、世界书、采用正文或 B3 Candidate Fact 提取内容。
2. 新内容一律进入 `candidate`；正文被采用不等于自动确认到 Canon。
3. 系统显示来源、版本、实体和可能冲突。
4. 用户可以编辑候选、确认、确认并锁定、废弃或创建 Retcon。
5. 确认只改变 C1 Ledger，不反向改写来源。
6. 同名或相似内容只参与搜索和冲突提示，绝不自动合并；合并必须由用户选择至少两条事实并产生一条保留来源的新事实。

## 3. Canon 冲突和 Retcon

冲突类型包括直接内容、时间、身份、状态、来源版本、适用时间不同、可能同时成立和信息不足。处理方式包括保留旧事实、采用新事实、设置生效时间、两者均保留、创建 Retcon、误报和暂缓。

Retcon 是独立历史记录：保存旧事实 ID、新事实 ID、原因、生效范围、受影响章节、角色和待复查来源。旧事实改为 `retconned`，新事实成为新的 `confirmed` 记录；历史条目不删除。锁定事实创建 Retcon 必须给出原因。Retcon 不自动同步正文和规划，待复查来源进入连续性问题。

## 4. 项目实体索引

实体类型为人物、地点、组织、物品、能力、世界规则、事件、秘密、剧情线、伏笔、章节和场景。本地索引读取角色卡、世界书、采用 B1/B2、采用正文、Canon、时间线、剧情线和伏笔的名称、别名、摘要与来源位置。

检索使用 Unicode NFKC 规范化、大小写折叠和多关键词包含匹配，支持类型、章节、来源和状态筛选。稳定来源类型加来源 ID 相同才复用实体；仅同名不会自动合并。第一版不依赖外部向量数据库。

## 5. 状态快照

- `CharacterSnapshot`：章节/场景、顺序、时间、地点、身体、情绪、目标、身份、阵营、已知信息、物品和未完成行动。
- `RelationshipSnapshot`：参与者、关系、信任、权力关系、未解决冲突和时间位置。
- `WorldSnapshot`：组织、社会、地点、规则或物品的当前状态与影响范围。

快照可以来自用户确认、B2 Scene Entry/Exit、B3 Candidate State Change、Canon、时间线或正文提取。只有用户确认来源才标记 `confirmed`；自动派生内容保持 `candidate`。同一顺序的角色地点或身体状态互斥时为确定性冲突。

## 6. 知情矩阵

`KnowledgeState` 分开记录信息本体、读者状态、公开/秘密、验证状态，以及每个角色的 `knows / believes_true / believes_false / suspects / does_not_know / forgot / unknown` 状态。角色条目同时保存获取时间、渠道、误解、怀疑和来源。

确定性检查包括公开与秘密同时成立、相互冲突状态；角色已知但缺少渠道、误解无来源、忘记关键事实属于需要人工判断的问题。作者知情不自动赋予角色或读者知情。

## 7. 剧情线和未解决问题

`PlotThread` 保存标题、角色、Plot Beat、章节、当前状态、下一节点、计划解决位置和事件。状态为候选、活跃、暂停、解决、废弃或 deprecated。事件记录引入、推进、复杂化、暂停、解决、重开和放弃。

`OpenQuestion` 区分未回答、部分回答、已回答、有意开放和废弃，并记录引入位置、计划回答位置和实际回答。活跃但无推进的剧情线与未安排回答的开放问题只标为启发式风险。

## 8. 伏笔设置与回收

`ForeshadowThread` 聚合设置、强化、误导、计划回收、实际回收和 Retcon 事件。事件带章节/场景、顺序、描述和来源。B2 Foreshadow Item 导入时保留稳定来源 ID。

确定性检查包括无设置回收、回收早于设置、废弃后仍使用；计划章节已完成仍未回收、长期未强化和匹配程度属于启发式提示。C1 是基础追踪器，不扩展为自动编排全书的伏笔数据库。

## 9. 全书时间线

时间线整合 B1 Story Timeline、B2 Chapter/Scene 时间、采用正文确认事件、Canon 时间事实、状态快照、剧情线事件和伏笔事件。时间表达支持明确日期、故事日、相对时间、顺序、范围和未知。

同一角色在同序事件的不同地点是确定冲突。日期/顺序、年龄、路程、伤势恢复和信息传播时间在信息足够时报告，信息不充分时降低置信度并标为启发式。

## 10. 摘要、偏差和过期

`ChapterSummary` 与 `SceneSummary` 保存采用正文版本 ID，以及事件、选择、状态、关系、信息、事实、剧情线、伏笔、问题和结束状态；每个摘要项标记事实、推断或建议。摘要产生的事实和状态仍是候选。

若摘要引用的 Draft Version 不再是 accepted 版本，摘要自动变为 `stale`。`PlanManuscriptDrift` 比较 B1 Plot Beat、B2 Chapter/Scene Plan 与采用正文，识别计划遗漏、重大新增、选择、信息、关系、时间、结束状态、覆盖和伏笔位置变化。用户只能显式标记有意偏差、采用正文变化、创建规划副本、创建修订任务或暂缓；系统不自动同步。

## 11. 连续性问题分类

检查覆盖 Canon 冲突、废弃事实使用、Retcon 未同步、身份/年龄/地点/身体/目标/关系、知情/秘密、时间/路程/伤势、剧情线、问题、伏笔、规划正文偏差、摘要/来源过期和锁定 Canon 违反。

严重程度沿用 A3：`critical / major / moderate / minor / note`；置信度为 `high / medium / low`。问题必须记录来源、章节、实体、最小修复、副作用、确定规则或启发式判断。处理状态为 `open / accepted / intentional / fixed / false_positive / deferred / retcon_required`。

## 12. 项目健康与写作进度

健康报告汇总正式字数、章节和场景完成度、Canon 冲突、严重问题、活跃/停滞剧情线、开放问题、待回收伏笔、过期摘要、偏差和待确认事实，并给出有限优先级建议。报告永不修改数据。

正式字数和完成度只统计 `accepted` Draft Version；模型生成但未接受的 alternative 或 incomplete 草稿不计入。进度支持卷、章、场景字数、规划/初稿/修订完成率、最近编辑和用户目标。

## 13. 下一章上下文包

上下文包从下一章目标、前章结束、活跃剧情线、最新人物/关系/知情状态、物品、未完成行动、伏笔/回收、禁止提前事件、相关 Canon、角色卡、世界书、语言/视角规则和当前风险构建。按权威、锁定和相关性在字符预算内裁剪，并向用户展示实际资料清单与截断提示。

用户可以编辑和锁定上下文包。发送到 B2/B3 只改变包的投递状态；接收方仍需创建新副本或草稿，不会修改当前采用规划或正文。

## 14. 版本、迁移和安全

C1 内部数据版本为 1，项目容器升级为 ProjectDraft v7。v1–v6 数据先按原 Schema 保留 A1–B3 字段，再为 C1 数组和选择 ID 添加安全默认值。迁移失败保留完整 `recoveryData`，不删除 localStorage 原值。

模型结构化请求支持 JSON 提取、Zod 校验、最多两次修复、超时、取消、中文错误和来源有效性标记。Prompt 版本为 `continuity-v1.0.0`。API 密钥仍只存在服务端环境变量中；C1 只保存 Provider 类型和模型名。
