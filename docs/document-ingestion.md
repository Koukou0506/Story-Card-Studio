# Phase C2.2 小说文件解析与逆向建模

## 1. 范围与安全默认值

C2.2 在一级导航“小说导入”中处理用户有权使用的 TXT 和 PDF 小说文件。目标是把原文转换为可追踪、可恢复、可逐项审查的候选数据，再复用现有 A1 角色卡、A2 世界书、B3 文风和 C1 连续性模型。解析结果不会直接覆盖项目中的已确认资料。

开始导入前必须确认文件处理权利。外部模型权限默认是 `local_only`；文件校验、解码或 PDF 文本层提取、清洗、分章、分块和确定性文体统计均可在本机完成。只有用户显式勾选外部模型权限时，来源权限才记录为 `chunks_only`；随后还要点击外部模型分析才会发出请求，每个请求只包含一个所选 `DocumentChunk`。系统不会把多个区块拼成整本小说发送，也不会自动改成按整章发送。

## 2. 支持的文件

| 类型 | 当前支持 | 处理边界 |
| --- | --- | --- |
| TXT | UTF-8、UTF-8 BOM、UTF-16 LE/BE（含 BOM 或启发式识别）、GB18030 | 自动检测置信度不足、替换字符过多或正文异常时，停止可信处理并要求用户查看预览、手动重选编码。不会把明显乱码静默当成成功结果。 |
| PDF | 具有可提取文本层的 PDF；逐页保存页码与字符范围 | 加密文件进入 `needs_password`；没有可用文本层的文件进入 `needs_ocr`。多栏、阅读顺序、空页和字符映射风险会形成警告，用户需要核对预览。 |

当前不支持 DOCX、EPUB、MOBI 或其他小说格式。文件必须同时通过扩展名、MIME 和签名检查；空文件、扩展名与内容不一致的文件、可执行文件特征和重复内容会被拒绝。内容指纹使用 SHA-256。

单文件默认上限为 50 MiB。服务调用方可通过 `maxBytes` 调整上限；调整时应同时评估浏览器内存、IndexedDB 配额、Provider 区块数量和工作区请求限制。该上限独立于手机项目 JSON 导入的 25 MiB 上限。

### PDF 与 OCR 的明确边界

`needs_ocr` 表示 PDF 没有可用文本层，解析流程在生成章节和候选前停止。C2.2 只定义了可替换的 OCR Adapter 接口，没有内置完整 OCR 引擎，也不会把空文本、图片文件名或模型猜测当成 OCR 结果。用户需要先在受信任工具中生成带文本层的 PDF，或以后接入经过明确授权的 OCR Adapter，再重新导入。

PDF 密码只用于当前解析调用，不进入 `DocumentSource`、`ProjectDraft`、文档资产、日志或导出文件。再次处理加密 PDF 时可能需要重新输入密码。

## 3. 工作区流程

“小说导入”按以下层次组织，手机端保持相同顺序并使用单列或分步呈现：

1. **上传**：选择 TXT/PDF，校验权利确认、类型、签名、大小和重复指纹。
2. **文本预览**：查看编码、PDF 页文本、警告和局部正文；编码可疑时先手动重选。
3. **章节确认**：确认自动分章结果，可重命名、拆分、合并、重排，或提供自定义标题正则。
4. **解析配置**：选择章节、分析深度和候选类型；外部模型默认关闭。
5. **处理进度**：按区块显示完成、失败、重试、取消和检查点状态。
6. **候选审查**：按角色、关系、世界设定、Canon、时间线、剧情线、伏笔、文风等类别查看来源证据和冲突。
7. **写入项目**：只把用户明确确认或编辑后确认的项目写成角色卡/世界书草稿、B3 文风资料或 C1 候选。
8. **解析报告**：汇总文件状态、警告、失败区块、候选决策、来源覆盖和仍需人工处理的限制。

用户可以只完成本地预处理并停在章节或候选审查阶段。取消长任务不会删除已完成区块和部分候选。

## 4. 处理管线

```text
权利确认与文件校验
  → 原始文件资产与 SHA-256 指纹
  → TXT 解码 / PDF 文本层逐页提取
  → 文本规范化与原始偏移映射
  → 章节识别与人工确认
  → 按章节、段落和句子边界分块
  → 本地统计 / 可选的外部区块提取
  → 区块结果去重与保守实体消歧
  → 章节及全书候选汇总
  → 用户逐项确认
  → 写入 A1 / A2 / B3 / C1 的草稿或候选
```

### 4.1 本地预处理

TXT 换行会统一为 `\n`。规范化会折叠部分空白、移除异常控制字符，并在可识别时清理重复页眉、页脚或纯页码；原始文件和原始文本仍单独保留。每次清洗都生成 normalized-to-raw 偏移段，来源跳转因此可以标记为精确、近似或无法映射，而不是伪造精确位置。

章节识别覆盖常见中文“第 X 章/卷/回/篇”、序章、楔子、前言、后记、番外和英文 Chapter/Prologue/Epilogue 等标题。识别是启发式结果；无标题正文会形成一个低置信度的“未分章内容”。人工重命名、拆分、合并和重排会记录为手动确认结果。

### 4.2 分块

分块先服从章节边界，再尽量在段落和句末标点处结束。默认目标大小为 6000 字符，相邻区块默认重叠 400 字符；配置允许根据设备和模型上下文调整。重叠区只用于保留上下文，合并时按规范化实体、内容和原始来源范围去重。

每个 `DocumentChunk` 保存章节、顺序、字符范围、估算 token、前后重叠长度、处理状态、重试次数、提取版本和至少一个 `SourceSpan`。区块失败不会使其他区块的成功结果失效。

### 4.3 本地与外部模型

本地流程不调用 Provider，可完成文件解析、章节/区块规划、来源映射和句长、段长、对话比例、标点、代词倾向、连接词及重复短语等确定性统计。

启用外部模型时，发送单位仍是用户选定章节中的单个区块，并携带来源范围和结构化输出要求。默认并发数为 2，最大并发数为 5。如果当前选择只有一个区块且该区块达到文档字符数约 90%，界面会阻止外部发送，要求减小目标区块或只选部分章节。`POST /api/analyze-document` 只接受严格的最小分块 DTO、可选 Provider/模型，以及 `depth`、人物范围和候选类别等受限分析开关；任务状态、失败详情、整本文本、密钥和其他配置字段均被拒绝。单块上限为 50,000 字符，总请求体上限为 256 KiB。接口强制 JSON、校验同源/工作区白名单和跨站请求标记、在工作区模式复用有效会话，并实施每来源频率限制。Provider 返回值必须经过 Schema 校验和区块内 Source Span 校验；模型填写的章节标题、摘录、映射状态和置信度不会被信任，而是替换为本地分块中的规范 Source Span。越界来源会被剔除，没有有效来源的候选会被丢弃。解析结果只进入候选集合。`ExternalModelPermission` 中的 `full_chapters` 是领域兼容值，当前 C2.2 工作流不会主动选择它，也不得借此绕过区块边界。

## 5. 来源追踪

每个候选尽可能保存一个或多个 `SourceSpan`：

- 文档 ID 与来源版本；
- 章节 ID 和标题；
- PDF 页码范围（可用时）；
- 段落范围（可用时）；
- 规范化文本的字符起止位置；
- 最长 280 字符的原文/规范化摘录；
- 提取置信度；
- `mapped`、`approximate` 或 `unmapped` 映射状态。

来源跳转会用文档、章、页、段和字符区间生成可读描述，并把文档来源转换为现有 B3/C1 可识别的 `document` 引用。清洗、页布局异常或缺失段落信息可能使定位变为近似；界面必须展示该状态，不能把近似范围描述成逐字精确证据。

来源版本变化后，旧候选可以保留用于审计，但应标为过期或要求重新核对。简短摘录用于审查，不代替原始文件。

## 6. 候选、消歧与写入

角色、别名、行为、关系、地点、组织、物品、能力、世界规则、事件、时间、秘密、剧情线、伏笔和文风结果首先都是候选。默认决策为 `pending`，支持确认、编辑后确认、合并为候选、保留候选、标记推断、忽略和冲突等显式决定。

实体消歧采取保守策略：

- 只有共享既有稳定实体 ID 时，服务才可判为 `same_entity`；
- 名称/别名和上下文高度一致时最多给出 `probably_same`，仍不代表用户确认；
- 同名但身份键冲突时标记 `conflict` 或不同实体；
- 单一场景中的临时行为保存在情境性表现中，不直接转换为永久性格。

转换服务只创建下游草稿或候选：

- Character Card 保持 `draft`，并在 extensions 中保存来源、权威和未确认状态；
- Lorebook 条目逐条保留 Source Span，写入前仍需审查；
- Canon、状态、时间线、剧情线、问题和伏笔进入 C1 候选，不直接成为确认 Canon；
- Style Profile 是 `alternative`，只抽象统计特征；
- Language Constraint 只生成 `preferred` 或 `advisory` 候选，不生成硬约束。

“写入项目”必须由用户明确触发。默认创建新草稿、比较或待合并候选，不覆盖已确认角色卡、世界书、正文、规划或 Canon，也不自动确认任何实体合并。

## 7. 任务、检查点与恢复

长任务按区块执行，任务记录包含进度、当前区块、完成/失败区块、重试上限和检查点。默认每个失败区块最多重试 2 次；限流、超时、网络或临时错误会使用短退避。重试耗尽后任务进入 `partially_completed` 或 `failed`，已经完成的结果仍可查看。

检查点保存完成区块 ID、失败区块 ID、最后区块顺序、阶段和保存时间。取消会保留已完成区块；恢复时跳过检查点中已经完成的区块，只继续未完成或用户选择重试的失败区块。检查点回调和 `ProjectDraft` 任务记录必须成功持久化后，才能把该位置视为跨刷新可恢复。

`ProjectDraft` 领域数据由 v7 迁移到 v8，新增 `documentIngestions[]` 和 `selectedDocumentIngestionId`，旧项目的 A1–C1 数据保持不变。迁移失败时创建可继续打开的安全空状态，并把原始输入放入 `recoveryData`；系统不会以空项目覆盖或删除原始恢复数据。

恢复来源包括：

- 任务检查点：恢复未完成的区块处理；
- `recoveryData`：导出或人工修复迁移失败的旧项目；
- 原始/原始文本资产：重新选择编码、重新分章或重新生成规范化文本；
- Project/ingestion JSON 导出：恢复 ingestion 元数据、任务、区块文本、候选和决策。

为支持检查点恢复和失败区块重跑，`DocumentIngestionProject` 会在 `chunks[].text` 中保存分块文本，ProjectDraft、完整项目 JSON 和 ingestion JSON 都会包含这些区块；全部区块合在一起可能覆盖大部分或全部提取正文，因此这些 JSON 必须按小说原稿同等敏感地保管。它们不包含原始二进制文件、独立的完整 raw/normalized 资产、PDF 密码或 Provider 密钥。只有 JSON 而缺少本机资产时，仍可查看候选并重跑已有区块，但不能重新解码原文件、保证重新分章或打开完整原文。

## 8. 存储、隐私与删除

原始文件、完整 raw 文本和完整 normalized 文本通过 `DocumentAssetStorage` 与常规 `ProjectDraft` 分离保存。`page_map` 是资产接口为后续实现预留的类型；当前 PDF 页范围直接保存在解析结果和 Source Span 中，不另写独立 page-map 资产。浏览器实现使用独立 IndexedDB 资产库；仅在运行环境没有 `IDBFactory` 时使用内存回退。IndexedDB 打开或操作失败会显式报错，不会在错误后静默丢到临时内存。用于检查点和 Provider 重跑的 `chunks[].text` 仍保存在 ProjectDraft；虽然采用分块结构，它们合在一起仍可能重复大部分或全部 normalized 正文，必须作为敏感正文处理。

隐私约束如下：

- `DocumentSource` 只保存元数据和资产引用，不保存完整正文或 PDF 密码；ProjectDraft 的 `chunks[].text` 例外地保存重跑所需区块；
- 本地模式默认不把任何正文发给 Provider；
- 外部模式只发送用户已允许且在发送清单中可见的区块；
- Provider 密钥仍只在服务端环境变量中，浏览器和项目数据不保存密钥；
- Service Worker/Cache Storage 不缓存小说资产、Provider 请求/响应、令牌或密码；
- 应用日志和错误日志不应包含完整小说、完整请求正文、PDF 密码、会话令牌或 Provider 密钥。项目/ingestion JSON 会包含 `chunks[].text`，导出界面必须明确提示其敏感性；密码、令牌、密钥和独立资产仍不会写入 JSON。

删除是不可恢复的数据操作，界面会先展示影响范围并要求确认。工作区提供两种不同语义：

- **删除原文件与提取文本，保留结构化候选**：按 `documentId` 删除 `original`、`raw_text`、`normalized_text` 以及以后可能存在的 `page_map` 资产，同时清空章节的 `paragraph.text` 和区块的 `text`，保留 Document Source 元数据、任务历史、候选、决策和短 Source Span 摘录。此后不能从应用内重新解析、重跑该区块或打开完整原文，来源跳转会显示资产不可用。
- **删除整个导入项目**：按该 ingestion 中的所有 `documentId` 删除全部资产，并从 ProjectDraft 移除整个 `DocumentIngestionProject` 及其章节、区块、任务、偏移映射和候选。

独立 IndexedDB 资产与 ProjectDraft 结构数据不是同一存储记录，因此删除流程必须明确执行对应两边的清理。已经由用户写入 A1/A2/B3/C1 的独立项目资料不会被级联删除；其中保留的来源引用会显示原文已删除或不可用。

建议在删除前导出项目备份。删除文本之前导出的 JSON 仍包含 `chunks[].text`，删除后重新导出的 JSON 才只保留候选和短摘录；两者都不包含原始二进制和完整 raw/normalized 资产。如需保留可重新解析的来源，用户还应自行保管原文件。浏览器站点数据被清除、无痕会话结束或存储配额回收后，本机资产也可能丢失。

## 9. 已知限制

- 没有内置完整 OCR；扫描 PDF 只会得到 `needs_ocr`。
- PDF.js 文本层不保证还原视觉阅读顺序。多栏、竖排、复杂脚注、表格、特殊字体、缺失字符映射和重复页眉页脚都可能需要人工核对。
- 编码识别是启发式过程；罕见编码、混合编码或已损坏 TXT 需要手动选择编码或先在外部工具转码。
- 当前实现会把文件读入内存后处理，不是流式超大文件解析；即使管理员提高 50 MiB 上限，也可能受设备内存和浏览器配额限制。
- 自动分章只覆盖常见标题模式，自定义排版和无标题作品需要手动调整。
- Source Span 在清洗或布局异常后可能是近似映射；短摘录上限为 280 字符。
- 实体消歧、语义提取、章节/全书汇总和文风解释均可能出错；候选确认和冲突处理不能省略。
- Style Profile 是统计与抽象特征，不承诺完美复刻作者文风；不得用于要求逐字模仿。
- 外部 Provider 模式可能产生费用并受其数据政策影响。启用前应检查发送清单，只选择确有必要的区块。
- Project/ingestion JSON 包含 `chunks[].text`，可能包含接近完整的提取正文，但不包含原始文档资产；若要重新解码、重新分章或验证 PDF 页布局，仍需要原文件或独立资产备份。
- 工作区服务器同步结构化 ProjectDraft，但浏览器 IndexedDB 中的原始二进制、raw/normalized 文本资产不会自动复制到其他设备；另一台设备仍可查看候选和已保存区块，但完整原文跳转或重新解析需要重新附加原文件。
