# Story Card Studio UI 审计（Phase C2.0）

## 1. 审计范围

本次审计覆盖：

- 应用外壳；
- 项目首页；
- 创意输入；
- 角色卡；
- 世界书；
- 剧情分析；
- 小说规划（含 B1/B2）；
- 正文写作；
- 连续性中心；
- 项目质量；
- 导入导出；
- 设置。

审计依据包括 `UI.md`、`README.md`、`docs/architecture.md`、全局 CSS、`src/app/page.tsx` 和现有全部页面组件。参考站与色卡的可验证结论记录在 [design-system.md](design-system.md)。

本轮没有修改生产 UI、CSS、业务逻辑或依赖。

## 2. 基线与技术现状

检查日期：2026-07-14。

技术栈：Next.js 16、React 19、TypeScript 5、Tailwind CSS 4 + 自定义 CSS、Zod 4、Vitest。

基线结果：

| 命令 | 结果 |
|---|---|
| `npm.cmd run typecheck` | 通过 |
| `npm.cmd test` | 30 个测试文件、207 个测试全部通过 |
| `npm.cmd run build` | 通过；10 个页面/路由成功生成或编译 |

当前 UI 由一个客户端主页持有全部一级工作区状态，使用 `activeTab` 在同一路由切换 9 个一级标签。领域与服务边界清晰，但页面层没有统一的视觉组件层，复杂工作区持续在同一个页面内嵌套标签、卡片和内联样式。

## 3. 可量化信号

### 3.1 导航与复杂度

- 一级导航有 9 个标签，全部横向排列在页面顶部；
- 小说规划内部有 8 个 B1 标签，B2 再有 7 个标签；
- 正文右侧检查器有 8 个标签；
- 连续性中心有 15 个横向标签；
- 当前没有独立项目首页，也没有独立设置页；
- 首页默认实际进入“创意输入”，而不是项目状态概览。

### 3.2 样式分散

| 组件 | 内联 `style` 次数 | 硬编码颜色 | 表单控件 | 按钮 |
|---|---:|---:|---:|---:|
| `ContinuityCenter` | 119 | 16 | 19 | 32 |
| `ProjectInput` | 28 | 0 | 20 | 0 |
| `QualityCheck` | 18 | 5 | 0 | 0 |
| `ProseWorkspace` | 16 | 2 | 22 | 38 |
| `LorebookWorkspace` | 11 | 0 | 33 | 19 |
| `ImportExport` | 10 | 1 | 1 | 2 |
| `GenerationPanel` | 9 | 3 | 2 | 2 |
| `CharacterEditor` | 9 | 1 | 4 | 0 |

`globals.css` 仍以 A1 的靛蓝 + Slate 颜色为中心，后续模块又增加绿色、红色、橙色、浅蓝和局部蓝色。`ContinuityCenter` 甚至自带一套 `#3451b2`、`#b42318` 和白色按钮规则，形成第三套视觉语言。

### 3.3 可访问性信号

代码中未发现：

- `:focus-visible`；
- `prefers-reduced-motion`；
- `aria-live`；
- `aria-current`；
- 标准 tabs / tabpanel 语义；
- 跳过导航链接；
- 使用 `htmlFor` 关联的表单标签。

这不代表所有页面不可用，但说明键盘、读屏、动态消息和低动效支持没有被系统化处理。

## 4. 全局问题

### 4.1 信息架构

1. **一级导航过载**：角色卡、世界书、分析、规划、正文、连续性和导入导出全部同权展示，没有“创作—规划—写作—维护”的任务分组。
2. **项目状态没有入口**：用户无法在进入具体工具前看到最近工作、写作进度、严重问题和下一步。
3. **工具设置散落**：Provider、模型、Style Profile、Language Constraint 和生成偏好分散在各页面，缺少稳定的项目设置入口。
4. **嵌套标签过深**：规划、正文和连续性把领域复杂度直接映射成横向标签，用户很难保持位置感。
5. **主操作不唯一**：同一工具条中经常并列新建、删除、生成、保存、导入、导出、发送 A3 等操作，没有明确优先级。

### 4.2 视觉一致性

1. A1 页面主要使用 `.card`、`.btn-*`、`.field-*`；C1 页面使用组件内 `box`、`button`、`input` style 对象；
2. 部分按钮没有任何共享 class，依赖浏览器默认样式；
3. Card 圆角从约 6px 到 10px 不等，表面色和边框也不统一；
4. 状态既用 Tag、左边框、浅底色，也用 Emoji，缺乏共同语法；
5. 空状态、加载状态和错误状态没有统一结构；
6. Emoji 被当作导航和标题图标，视觉重量、平台渲染和无障碍名称都不稳定。

### 4.3 信息层级

1. 页面缺少稳定的 H1 + 页面说明 + 状态 + 主操作结构；
2. `.card-header` 多数只有 16px，与页面标题、模块标题的差距不足；
3. 大量 12px、11.2px 的提示文字承担关键版本、来源和保护说明；
4. 表单字段、来源资料、生成控制和结果报告常被放在相同卡片层级；
5. 高风险操作经常与日常操作同排，危险程度主要依靠红色表示；
6. 长报告没有目录、锚点或粘性过滤，扫描成本高。

### 4.4 间距与页面密度

1. 页面主间距集中在 8–16px，复杂区域几乎没有 32–64px 的章节留白；
2. 卡片统一 20px padding，但卡片内部继续嵌套卡片，形成“边框套边框”；
3. 很多字段用两列网格硬塞进卡片，字段名称长时会显得拥挤；
4. `button-row` 在高密度页面频繁换行，主要操作位置随内容变化；
5. 连续性中心信息虽多，但横向按钮和内联表单没有密度层级；
6. 正文写作三列固定为 240px / 主列 / 360px，在中等屏幕会直接挤压编辑器。

### 4.5 字号与排版

1. 应用标题只有 20px，一级页面通常没有显著 H1；
2. Field label 为 12px，Field hint 为 11.2px，长时间阅读和高 DPI 环境下偏小；
3. 正文编辑器继承 14px 表单字号，不适合长篇写作；
4. 中文正文、英文 ID、JSON 和元数据没有分别使用阅读字体、UI 字体和等宽字体；
5. H2/H3 未形成一致的字号与上下间距系统；
6. 报告中高价值结论与低价值元数据常以相同 14px 呈现。

### 4.6 色彩

1. 现有 `#6366f1` 靛蓝与目标钴蓝 `#294A97` 不一致；
2. `#94a3b8` 等浅灰被用于 11–12px 文字，存在对比度风险；
3. Error、warning、success 和 info 在不同页面使用不同色值；
4. 浅色边框承担输入框边界，部分状态只靠边框颜色；
5. 当前白色卡片 + 冷灰背景偏通用 SaaS，与暖米白编辑工作台定位不一致；
6. C1 组件硬编码白色和冷蓝，无法随全局主题 Token 一致变化。

### 4.7 重复组件

重复但未统一的模式包括：

- 页面头部与项目选择器；
- Provider / Model 选择；
- 新建 / 复制 / 删除工具条；
- 筛选条；
- Context source row；
- Issue card；
- Source reference；
- Status tag；
- Empty / Loading / Error state；
- Import / Export 按钮组；
- Version switcher 和 JSON `pre` 比较；
- Confirm、Toast 和 Inline notice。

这些重复不应通过复制 CSS 继续扩张，应该在 C2.0 实施阶段收敛到共享视觉组件，但保持现有服务调用与数据接口。

### 4.8 可访问性

高优先级风险：

- 全局没有可见的键盘焦点标准；
- `.collapsible-toggle` 使用 `div onClick`，不是原生按钮；
- 多数表单“标签”是 `div`，点击标签不能聚焦输入；
- Toast 自动消失且没有 live region；
- 标签导航没有角色、选中状态或方向键行为；
- 状态使用 🟢、⚪、⚠️、Emoji 但没有稳定文本语义；
- 多处 `<select multiple>` 和密集复选框没有清楚的分组说明；
- 错误信息没有与具体字段关联；
- 动画没有 reduced-motion 处理；
- 多个按钮文字使用英文内部枚举，降低中文界面可理解性。

### 4.9 响应式风险

1. 只有少数 `max-width: 1024px` 规则，缺少 768px 以下基础；
2. 顶部 9 标签没有溢出策略；
3. Continuity 的 15 标签只能横向滚动，当前项容易离开视口；
4. Prose 固定三列没有断点；
5. Project Input 的两列字段和高级设置没有移动端降列；
6. 表格、版本 JSON、长来源 ID 和多按钮操作容易横向溢出；
7. 触控目标普遍小于 44px；
8. 固定高度和 max-height 列表在平板横屏容易形成多个嵌套滚动区。

## 5. 逐页审计

## 5.1 应用外壳

**当前结构**

- 顶部品牌标题、Phase 副标题、本地草稿版本、质量检查和清除数据；
- 下方 9 个横向一级标签；
- 主内容在单一路由内切换；
- 全局迁移错误和右上 Toast。

**可以保留**

- `useDraft` 的统一项目状态；
- 一级工作区切换语义；
- 本地草稿版本和迁移恢复入口；
- 全局 Toast 的行为意图。

**需要重构**

- 顶部标签改为按工作流分组的左侧导航；
- 顶栏只承载项目上下文、保存状态、搜索/命令与页面级操作；
- 新增独立项目首页和设置入口；
- 清除数据移入项目设置的危险区；
- 页面标题从各 Card 内提升到统一 `PageHeader`；
- 增加跳过导航、焦点管理、`aria-current` 和响应式抽屉。

## 5.2 项目首页

**当前结构**

没有独立项目首页。默认“创意输入”同时显示生成控制、项目输入和角色卡预览。

**问题**

- 旧项目打开后仍先看到角色生成，不符合 A1–C1 全流程；
- 无法快速判断最近编辑、写作进度、严重问题、未处理冲突和下一步；
- 新用户与已有项目用户共享同一个高密度入口；
- 生成角色卡被误当成整个应用的唯一主任务。

**需要新增的 UI 层**

- 项目标题和本地状态；
- “继续写作”主 CTA；
- 最近工作、项目健康、进度和下一步；
- 新项目空状态与导入入口；
- 快速进入角色卡、世界书、规划、正文和连续性的卡片。

这只是新的视觉与导航入口，不创建第二份项目数据。

## 5.3 创意输入

**当前结构**

- 左列：Generation Panel + Project Input；
- 右列：已有角色卡编辑或空状态；
- Project Input 包含 6 个基础字段和可展开的 11 个高级字段；
- Provider 与模型固定显示在生成卡片顶部。

**问题**

- 生成控制先于创意本身，视觉顺序倒置；
- 原始想法与辅助字段没有分组为“必须 / 可选 / 约束”；
- 高级区使用可点击 `div`，无键盘语义；
- 两列网格在窄屏没有专属断点；
- 每个字段都使用相同层级，用户难以先完成最小输入；
- Provider 选择占用主流程空间。

**可以保留**

- 原始想法永不被覆盖的业务说明；
- 原创/同人模式；
- 高级字段的折叠意图；
- 生成、取消和错误处理。

**需要重构**

- 先呈现项目名称、模式和原始想法；
- “角色方向”“场景与关系”“创作约束”分组；
- Provider/模型进入生成按钮旁的次级设置 Popover；
- 右侧预览改为非阻塞摘要，不直接嵌完整角色编辑器。

## 5.4 角色卡

**当前结构**

- 左列仅有角色生成面板；
- 右列为单张角色卡长表单；
- 基础字段顺序展开，高级字段统一折叠；
- extensions 只读提示位于表单底部。

**问题**

- 已有角色卡编辑时，生成面板仍长期占据约一半页面；
- 没有角色列表、字段目录、保存状态或质量检查侧栏；
- `description`、`personality`、`scenario`、`first_mes`、`mes_example` 等长字段连续堆叠；
- 关键提示使用 11.2px；
- 标签、版本和扩展信息混在同一高级折叠区；
- 质量检查需要离开页面到一级标签。

**可以保留**

- `FIELD_DEFINITIONS` 数据驱动字段定义；
- tags / greetings 的转换行为；
- extensions 自动保留说明；
- 字段更新接口。

**需要重构**

- 左侧角色/版本列表，中部编辑器，右侧质量与来源检查器；
- 按“基础、人格、互动、提示、元数据”分节；
- 生成改为页面主操作或局部动作，不固定占列；
- 编辑区使用 720–840px 可读列和粘性保存状态。

## 5.5 世界书

**当前结构**

- 左侧：世界书总览、统计、新建/复制/删除、生成配置；
- 右侧：条目筛选、批量操作、折叠条目编辑；
- 下方全宽：质量检查 + 激活模拟；
- 最下方全宽：导入导出与 Character Book 写回。

**优点**

- 功能范围完整，主从关系已经存在；
- 条目搜索、分类、批量启停和高级规则明确；
- 未知字段与格式专属字段有可见说明；
- 删除和写回具有业务保护。

**问题**

- 四个工作阶段在同一长页面纵向堆叠；
- 左侧生成面板和书籍元数据争夺注意力；
- 条目折叠卡片内有大量输入、按钮和警告，扫描效率低；
- 展开条目时列表位置不稳定；
- 质量、模拟和导入导出离当前条目很远；
- “启用状态”依靠圆点 Emoji；
- 高级格式数据以 JSON `pre` 展示，缺少格式和只读层级。

**可以保留**

- Lorebook / Entry 的现有增删改查；
- 搜索、分类、批量操作和排序；
- 质量检查与模拟服务；
- Character Book 合并预览和安全写回。

**需要重构**

- 三段主从布局：书籍列表 / 条目列表 / 条目检查器；
- 条目列表支持舒适卡片和紧凑表格模式；
- 质量、模拟、格式和导入导出进入右侧检查器或页面级抽屉；
- 批量操作使用选中后出现的上下文工具条；
- 写回角色卡继续使用明确确认对话框。

## 5.6 剧情分析

**当前结构**

- 左侧长输入表单；
- 右侧上下文选择、Token 预算和 Provider；
- 分析结果在两列下方全宽展开；
- 报告含摘要、评分、问题、角色、因果、关系、连续性、分支和来源。

**问题**

- 输入字段多但缺少步骤或内部目录；
- 上下文清单默认完全展开，容易盖过主操作；
- 报告长而无粘性目录；
- 过滤器只使用内部英文枚举；
- 问题详情、来源和建议在卡片内继续嵌套；
- “确认保存”和导出操作与报告标题挤在同一行；
- 评分视觉重量容易超过综合结论和 critical 问题。

**可以保留**

- 输入与上下文并列的结构；
- 来源有效性、权威和 Token 预算；
- 报告筛选、复制建议和保存备注；
- 结果不会覆盖原输入的保护。

**需要重构**

- 输入按“剧情、状态、约束、分支”分节；
- 上下文改为摘要 + 可展开抽屉；
- 报告使用左侧章节目录或顶部锚点；
- 先展示一句结论、critical 问题和最低成本修复，再展示评分；
- 枚举提供中文显示文本。

## 5.7 小说规划

**当前结构**

- 顶部全宽项目/版本/生成工具条；
- 左侧生成与上下文；
- 右侧 B1 规划的 8 标签；
- “章节”标签中再嵌入 B2 的 7 标签；
- 大纲、场景等编辑器使用卡片 + 多字段网格。

**问题**

- 两层横向标签让 B1/B2 位置关系不清；
- 英文标签与中文页面混用；
- 工具条同时放项目、版本、生成、采用和删除；
- 大量内部 ID 直接要求用户输入；
- 版本比较直接显示 JSON；
- 章节/场景字段数高，但没有字段分组和粘性目录；
- 普通按钮、主按钮和危险按钮风格不统一。

**可以保留**

- B1/B2 通过稳定 ID 关联；
- 版本、锁定、生成草稿和确认保存机制；
- 规划上下文与 A3 集成；
- Volume/Chapter/Scene 的既有业务组件和校验结果。

**需要重构**

- 小说规划内使用左侧模块导航：故事圣经、角色、角色弧、关系、宏观大纲、分卷、章节、场景、时间线；
- 项目和版本选择进入 Page Header；
- 当前对象的编辑器居中，问题/来源/上下文在右侧检查器；
- ID 输入改为可搜索实体选择器；
- 版本比较改为字段级和列表级差异。

## 5.8 正文写作

**当前结构**

- 顶部正文项目、导入导出和删除；
- 固定三列：240px 章节场景导航、主编辑器、360px 检查器；
- 主列下方固定生成与局部修订；
- 右侧 8 标签：计划、上下文、检查、事实、状态、修订、版本、风格；
- 正文为一个 500px 高的 `textarea`；
- 段落锁定以正文下方按钮列表呈现。

**优点**

- 已经具备典型写作工作台的三栏雏形；
- 原稿、版本、Revision 和范围保护清楚；
- 计划、质量、候选和版本放在侧栏的方向正确；
- 生成只创建备选版本。

**问题**

- 固定三列无断点，中等桌面会压缩正文；
- 顶部导入导出和删除分散写作注意力；
- `textarea` 默认 14px，行宽随主列无限变化；
- 生成控制永久占据正文下方，打断写作流；
- 段落锁定与正文位置分离，用户无法直观看到哪段被锁；
- 右侧 8 标签挤成一排；
- Diff 使用红/绿整块背景，层级和色觉适配不足；
- Style 与 Language 设置挤在同一超长面板。

**可以保留**

- 章节/场景导航、版本切换和 word count；
- 生成模式、Edit Scope、取消和临时草稿；
- 右侧检查器中的数据内容；
- Revision、候选事实、候选状态和 B2 副本保护。

**需要重构**

- 中央正文列固定在约 760px 并提供专注模式；
- 右侧检查器改为分组导航：计划、生成、检查、版本；低频项放二级列表；
- 生成控制改为右侧面板或底部可收起 Composer；
- 段落锁定直接显示在段落边缘；
- 导入导出和删除移入溢出菜单；
- Diff 使用行级/段落级标记、图标和文字，不只靠红绿。

## 5.9 连续性中心

**当前结构**

- 顶部项目选择、刷新、导入导出、删除；
- 15 个横向视图按钮；
- 视图内使用统计卡、表单、表格和长列表；
- 几乎全部样式由组件内对象控制。

**问题**

- 15 个标签已经超过横向导航可管理范围；
- 119 处内联样式使主题、密度、焦点和响应式无法统一；
- 页面有独立的蓝/红/白配色；
- Canon、实体、状态、知情、剧情线、伏笔、时间线和问题没有更高层分组；
- 表格缺少窄屏降级；
- 部分操作直接使用 `prompt()`；
- 统计卡数量多但缺少优先顺序；
- 当前刷新、删除、导出与日常浏览同权。

**可以保留**

- 15 个业务视图的数据和操作；
- 本地索引、筛选、刷新和上下文预览；
- Canon 冲突、Retcon、Issue 和下一章包流程；
- Stat、Sources、IssueCard 的内容结构。

**需要重构**

- 左侧分组导航：概览；Canon/实体；状态/知情；剧情线/问题/伏笔；时间线/摘要/偏差；连续性/检索/进度/下一章；
- 共享 Stat、SourceReference、IssueRow 和 DataTable；
- Inline style 全部迁移到 Token 化样式；
- `prompt()` 改为可撤销对话框；
- 顶部只保留“刷新索引”主操作，导出/删除进入次级菜单。

## 5.10 项目质量

**当前结构**

- 独立一级标签只显示角色卡质量报告；
- 世界书、规划、正文和连续性质量检查都留在各自页面。

**问题**

- “项目质量”命名与实际只检查角色卡不一致；
- 一级导航成本过高；
- Issue 展示与 A3/B1/B2/B3/C1 各自的 Issue 卡不一致；
- 空状态指向一个并不存在于当前页面的“检查质量”按钮。

**建议**

- 角色卡质量移入角色卡检查器；
- 项目首页显示跨模块健康摘要并链接到来源页；
- 如果保留一级“质量”，它应成为只读聚合器，而不是角色卡专页。

## 5.11 导入导出

**当前结构**

- 左侧 Character Card V2 导入导出；
- 右侧卡片提示世界书操作位于世界书工作区；
- 成功/错误消息复用 Toast class 但改为相对定位。

**问题**

- 页面只覆盖角色卡，其他导入导出分散；
- 文件格式、影响对象和 round-trip 信息没有统一表格；
- 相同 Toast 组件同时被当作浮层和内联通知；
- 导出是主按钮、导入是次按钮，但在恢复项目时优先级可能相反；
- 没有项目级备份总览。

**可以保留**

- 现有导入导出服务与安全文件名；
- 文件类型校验和成功/错误信息；
- 各业务工作区内部的上下文导入导出入口。

**需要重构**

- 形成按“项目备份、角色卡、世界书、规划、正文、连续性”分组的格式中心；
- 每种格式显示来源、目标、是否可无损 round-trip 和警告；
- 上传区使用明确文件选择与拖放视觉，但不改变解析逻辑。

## 5.12 设置

**当前结构**

没有独立设置页。Provider/Model 分散在 Generation Panel、世界书和剧情分析；规划、正文使用项目偏好；Style Profile 和 Language Constraint 位于正文侧栏。

**问题**

- 用户难以理解项目级设置与单次生成设置的差异；
- Provider 选项重复定义；
- API 密钥只在环境变量中的安全事实没有集中解释；
- 视觉密度、默认字体、正文阅读字体和 reduced motion 无入口；
- 危险的“清除数据”反而位于全局 Header。

**建议结构**

- 项目：名称、默认 Provider/Model、自动保存、数据版本和备份；
- 写作：默认 Style Profile、Language Constraint、正文阅读字体；
- 界面：舒适/紧凑密度、动效偏好；
- 隐私：哪些内容会发送给模型、密钥仅服务端读取；
- 危险区：清除本地项目，必须二次确认并提供备份入口。

## 6. 可以保留的组件与行为

以下内容适合“换壳不换逻辑”：

- `useDraft`、迁移和 recovery 行为；
- `CharacterEditor` 的字段定义与字段值转换；
- `GenerationPanel` 的 Provider/Model、生成和取消行为；
- Lorebook 的条目 CRUD、筛选、模拟、质量与安全写回；
- Analysis 的 Context、Report、Issue、Source 数据；
- B1/B2 的版本、锁定、生成草稿和 A3 集成；
- Prose 的 Edit Scope、Revision、Diff、Candidate 和版本保护；
- Continuity 的 Canon、Snapshot、Thread、Foreshadow、Timeline 和 Issue 操作；
- 所有导入导出服务和文件安全处理。

## 7. 需要重构的视觉组件

第一优先级：

- App Shell 和导航；
- Page Header；
- Button、Field、Card、Tabs、Toolbar；
- Empty/Loading/Error/Toast；
- Focus、键盘和动效规则。

第二优先级：

- Split Pane、Inspector、Context Drawer；
- Issue Row、Source Reference、Status Badge；
- Version Switcher、Diff Viewer；
- Data Table 和主从列表；
- Confirm Dialog 和安全写回摘要。

第三优先级：

- 复杂筛选器；
- 命令面板/全局检索；
- 密度切换；
- 平板抽屉和 C2.1 移动端交互。

## 8. 审计结论

当前应用的核心问题不是功能不足，而是 Phase A1–C1 的功能持续叠加在 A1 视觉壳上：一级导航过载、页面标题不成体系、复杂工作区标签过深、样式重复且不可访问性规则缺失。最适合的改造不是把所有页面做成同一种大卡片，也不是把所有内容压成专业表格，而是建立共享外壳和分层密度系统，优先改造五个关键页面，再迁移第二批页面。
