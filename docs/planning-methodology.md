# 小说规划方法论

## 故事圣经

故事圣经保存跨版本稳定的创作基线：一句话故事、梗概、类型、基调、主题、核心命题、叙事视角、时间范围、地点、世界规则、核心冲突、主角目标、对抗力量、风险、代价、结局方向、不可修改条件、禁止方向和未解决问题。每个字段可独立锁定；锁定值的权威级别最高，模型生成只能读取，不能覆盖。

## 角色规划与角色弧

`CharacterPlan` 通过 `linkedCharacterCardId` 关联现有角色卡，不复制出失联角色。角色规划记录故事功能、初始状态、外在目标、内在需求、欲望、恐惧、错误信念、优势、弱点、行为边界、关键选择、变化方向和结局状态。

`CharacterArc` 通过 Beat ID 关联触发事件、第一次主动选择、压力升级、中段变化、最大失败和最终选择。支持正向、负向、平稳、悲剧、开放与自定义弧。声明的变化必须由 Plot Beat 的 `CharacterStateChange` 支持。

## 情节节点与因果关系

宏观大纲支持自由、三幕、四幕和自定义结构。Plot Beat 记录前置条件、触发原因、主要行动、直接结果、长期后果、风险、代价、信息变化和三类状态变化。

依赖类型：

- `causes`：直接导致；`enables`：提供条件；`motivates`：形成动机；
- `reveals`：揭示信息；`prevents`：阻止；`complicates`：增加困难；
- `resolves`：解决；`foreshadows`：铺垫；`contradicts`：互相冲突。

必要原因、条件和信息必须排在结果之前。自依赖、不存在的节点和有向循环均视为错误。

## 时间线与状态变化

时间支持明确日期、故事日、相对事件和仅确定顺序。未知时间不自动补成精确日期。第一版检查故事日倒序、同一角色同日跨地点以及相对事件引用。

状态采用结构化文字：

- `CharacterStateChange`：角色、前后状态、原因、是否公开；
- `RelationshipStateChange`：参与角色、前后关系、触发和余波；
- `WorldStateChange`：实体、前后状态、原因和影响范围。

检查器按情节顺序传递“变化后”状态；下一节点声明的“变化前”状态不一致时报告连续性问题。

## 内容状态和来源

内容状态为 `locked`、`confirmed`、`draft`、`suggested`、`deprecated`、`unknown`。来源引用包含来源类型、ID、名称、字段、摘录和版本。只有本次 Planning Context 中实际包含且类型、ID、版本匹配的来源才有效。

## 多版本与比较

`StoryPlan` 保存多个 `OutlineVariant`，并分别记录父版本、创建来源、状态、是否采用和时间。新生成、替代版本和 A3 修订均创建副本，不覆盖旧版本。

版本比较显示故事圣经字段、角色弧、节点新增/删除/修改、结局、新增设定、冲突风险、额外铺垫和粗略修改成本。修改成本只用于比较工作量，不是质量排名。

## 一致性检查

固定检查：核心冲突、主角目标、风险代价、节点触发、中间行动、角色动机、角色弧事件、声明状态变化、关系触发、时间顺序、地点路程、信息获得顺序、角色/关系/世界状态、世界规则、重大事件后果、高潮关联、结局回应、新设定标记、锁定保护、来源有效性、重复内容和因果循环。

每项问题包含严重程度、置信度、相关节点/角色、依据、最小修改和副作用。检查器只报告，不自动修改规划。
## B1 实施补充：上下文、锁定与来源

规划生成前由 `PlanningContextBuilder` 按权威等级排序并执行 token 预算：原始创意和锁定约束优先，其次是现有规划、角色卡、相关世界书条目和用户选定的 A3 报告。世界书只按选定书籍、启用状态和创意/目标关键词筛选；常驻条目可直接纳入。上下文清单会显示来源 ID、版本、权威等级、是否锁定、是否发送以及裁剪原因。

来源引用区分 `confirmed_fact`、`source_setting`、`user_assumption`、`model_inference`、`model_suggestion` 和 `unknown`，并用 `inference` 标记跨来源推断。生成后必须按来源类型、实体 ID 和版本重新校验；无效引用保留为无效状态并产生一致性问题，不作为有效证据展示。

锁定规则分三层：故事圣经的 `lockedFields`、带 `locked` 状态的内容实体、以及锁定创作约束。局部生成只替换选定模块，合并器会恢复锁定实体，即使 Provider 没有回传该实体也不得删除。所有生成结果先作为草稿，用户确认后才保存为新版本。

## 版本比较规则

比较两个 `OutlineVariant` 时同时报告故事圣经字段、角色弧、关系路线、节点新增/删除/修改、结局方向、新增设定、严重冲突、额外铺垫和粗略修改成本。结果是差异清单而非单一排名；`adopted` 只表示用户当前采用的版本，`deprecated` 仍保留以便恢复和追溯。

## 一致性检查边界

第一版检查依赖节点、自我依赖和循环、原因与信息顺序、重大事件结果、角色弧与节点关联、关系触发、时间线相对引用、同日跨地点、角色/关系/世界状态连续性、世界规则、锁定修改、来源有效性、高潮关联和重复内容。检查结果包含严重程度、置信度、相关节点/角色、最小修改和副作用；检查器不自动改写故事规划。
